> **覆盖规则**：
> - 每个 AC 必须至少有一个测试用例覆盖

> - **交互类 AC**：至少 1 个 E2E 交互测试
> - **性能类 AC**：如无自动化性能测试，标注"人工验证"并在降级方案中说明

## 2. E2E 测试用例

> 覆盖前端页面的完整交互流程。
> 本项目当前未配置 E2E 测试框架，以下用例降级为人工验收。见降级方案。

### 2.1 冒烟测试（batch-a）

> 目标：验证页面可达、数据加载。每个新增页面必须有冒烟测试。

| 用例ID | 用例名称 | 对应 AC | 测试步骤 | 预期结果 |
|--------|----------|---------|----------|----------|
| E2E-001 | {页面名}页面可达 | AC-001 | 1. 访问 {URL}<br>2. 等待页面加载 | 页面正确渲染，无 404 |
| E2E-002 | {功能名}数据加载 | AC-001 | 1. 进入 {页面}<br>2. 等待接口返回 | 至少一个 API 请求成功，数据渲染到页面（非占位符） |

**三层验证标准**（每个冒烟测试必须满足）：
1. **页面可达**：URL 正确，主容器可见
2. **数据加载**：至少一个 API 请求返回成功
3. **数据渲染**：至少一个数据驱动的 DOM 元素包含非空文本（非占位符）

**全局错误监听（必须内置）**：

> 全局错误监听代码片段详见 `rules/testing.md` 🔴 "全局错误监听" 节的强制代码模板。

### 2.3 权限测试（batch-c）

> 目标：验证权限控制。每个"权限类"AC 必须有权限测试。
> 权限测试是安全底线，不可省略。

| 用例ID | 用例名称 | 对应 AC | 测试步骤 | 预期结果 |
|--------|----------|---------|----------|----------|
| E2E-009 | {角色}可访问{功能} | AC-003 | 1. 使用{角色}账号登录<br>2. 访问{页面} | 页面正常显示，功能可用 |
| E2E-010 | {角色}无权限访问{功能} | AC-003 | 1. 使用{角色}账号登录<br>2. 访问{页面} | 返回 403 或页面不显示 |
| E2E-011 | 横向越权检测-IDOR | AC-003 | 1. 用户 A 创建{资源}<br>2. 用户 B 尝试访问/修改用户 A 的{资源} | 拒绝访问，返回 403 |
| E2E-012 | 前端按钮权限-无权限用户 | AC-003 | 1. 使用无权限账号登录<br>2. 查看{页面} | 操作按钮不显示 |

**测试数据覆盖关键角色**：
- 超级管理员（admin）— 全部权限
- 普通用户（user）— 部分权限
- 无权限用户（guest）— 无此功能权限
- 无数据用户（empty）— 有权限但无数据（测试空状态）

> **权限测试覆盖检查**：
> - 每个"权限类"AC 是否有正向（有权）+ 反向（无权）测试？
> - 是否有 IDOR 横向越权测试？（写操作必须校验直属关系）
> - 是否测试了前端按钮的显隐？（有权限显示、无权限隐藏）
> - 是否测试了直接调用 API 绕过前端的场景？（后端必须独立校验）

## 3. 测试数据规范

### 3.1 Seed 数据

| 数据类型 | 准备方式 | 说明 |
|---------|---------|------|
| 角色数据 | Seed 脚本 | admin、user、guest、empty 各 1 个 |
| 业务数据 | Seed 脚本 | 每种状态至少 1 条 |
| 边界数据 | 测试用例内准备 | 最大值、最小值、空值 |
| 冲突数据 | Seed 脚本 | 用于测试并发/冲突场景 |

### 3.2 测试数据示例

```sql
SET NAMES utf8mb4;
SET CHARACTER SET utf8mb4;
-- DB_TYPE: MySQL 8.0

-- Seed 脚本（可重复执行，幂等）
INSERT INTO sys_user (id, username, role, ...) VALUES (1, 'admin', 'admin', ...) ON DUPLICATE KEY UPDATE ...;
INSERT INTO sys_user (id, username, role, ...) VALUES (2, 'user', 'user', ...) ON DUPLICATE KEY UPDATE ...;
INSERT INTO sys_user (id, username, role, ...) VALUES (3, 'guest', 'guest', ...) ON DUPLICATE KEY UPDATE ...;
INSERT INTO sys_user (id, username, role, ...) VALUES (4, 'empty', 'user', ...) ON DUPLICATE KEY UPDATE ...;
```

### 3.3 测试账号

| 账号 | 角色 | 用途 | 数据状态 |
|------|------|------|---------|
| admin | 超级管理员 | 全权限测试 | 有业务数据 |
| user | 普通用户 | 部分权限测试 | 有业务数据 |
| guest | 无权限用户 | 权限边界测试 | 无此功能数据 |
| empty | 普通用户（无数据） | 空状态测试 | 无业务数据 |

> **注意**：不要只测有数据的用户。empty 账号用于验证空状态和 nil 兜底。
