# /sds-codereview Skill 规范

## 触发场景

- 用户主动要求审查特定代码变更
- `sds-dev` 流程外需要独立代码审查
- 代码合并前需要质量把关
- 安全审计需要

## 执行方式

`/sds-codereview` 是对 `sda-code-reviewer` agent 的包装 Skill。

执行时，主 CC 直接调用 `sda-code-reviewer` agent 进行审查。

## 审查范围

默认审查当前变更（未提交的修改）。如需审查特定文件，在命令中指定：

```
/sds-codereview                              # 审查当前所有未提交变更
/sds-codereview yudao-module-system/         # 审查指定目录
/sds-codereview src/main/java/               # 审查指定路径
```

## 前置条件

调用前，主 CC 应先收集以下信息：

| 信息 | 获取方式 | 必需 |
|------|----------|------|
| 变更文件列表 | `git diff --name-only` 或用户指定 | 是 |
| 变更内容 | `git diff` | 是 |
| 审查范围说明 | 新增功能 / Bug 修复 / 重构 | 否 |
| 相关需求文档 | `.claude/specs/{name}/requirements.md` | 否 |

## 工作流程

```
确定审查范围 → 收集变更信息 → 调用 sda-code-reviewer → 输出审查报告 → 跟踪修复
```

### 详细步骤

1. **主 CC 接收 `/sds-codereview` 命令**
2. **确定审查范围**
   - 默认：当前未提交的变更（`git diff`）
   - 指定路径：用户指定的文件或目录
   - 全量：整个模块的代码
3. **收集变更信息**
   - 变更文件列表
   - 变更内容（diff）
   - 变更类型（新增/修改/删除）
4. **调用 `sda-code-reviewer` agent**
   - 传递变更信息和审查范围
   - agent 执行五维度审查
5. **输出审查报告**
   - P0/P1/P2 问题分级
   - 具体修复建议
6. **跟踪修复**
   - 如有 P0 问题：必须修复后重新审查
   - 如有 P1 问题：建议修复，由用户决定
   - P2 问题：可选修复

## 审查维度

| 维度 | 关注点 |
|------|--------|
| 质量 | 业务逻辑正确性、错误处理完善度、代码健壮性 |
| 安全 | SQL 注入、XSS、敏感信息、权限控制 |
| 可维护性 | 命名规范、代码结构、注释文档、代码风格 |
| 性能 | 数据库性能、缓存策略、内存资源、算法复杂度 |
| 测试覆盖 | 测试存在性、测试有效性、覆盖率要求 |

## 审查通过条件

| 条件 | 要求 |
|------|------|
| P0 问题 | 0 个（必须全部修复） |
| P1 问题 | ≤ 3 个（或已确认延后修复） |
| P2 问题 | 不限 |
| 安全检查 | 全部通过 |

## 输出

直接输出 `sda-code-reviewer` 的 `## Code Review Report` 格式：

```markdown
## Code Review Report

### 审查概要
- 审查范围：[文件数量/模块]
- 审查时间：[时间]
- 审查结论：[通过/有条件通过/不通过]

### P0 问题（必须修复）
| 文件 | 行号 | 问题描述 | 修复建议 |
|------|------|----------|----------|

### P1 问题（建议修复）
| 文件 | 行号 | 问题描述 | 修复建议 |
|------|------|----------|----------|

### P2 问题（可选优化）
| 文件 | 行号 | 问题描述 | 修复建议 |
|------|------|----------|----------|

### 审查通过条件
- [ ] P0 问题数量：0
- [ ] P1 问题数量：≤ 3
- [ ] 安全检查：全部通过
```

## 注意事项

- 安全问题一律为 P0，不可降级
- 审查结果需关联具体文件和行号
- 修复建议要具体可执行，避免模糊描述
- 如 P0 问题超过 5 个，建议先修复再重新审查
- 审查不改变代码，只输出报告和修复建议
