# /sdc-plan 命令规范

## 用途

在执行任何高风险操作前，先输出结构化的 Plan，经用户确认后再执行。确保变更可控、风险可评估。

## 触发条件

与 CLAUDE.md「必须先 Plan 的高风险操作」一致：

| 触发条件 | 示例 |
|----------|------|
| 修改 3 个及以上文件 | 新增功能涉及前后端+数据库 |
| 代码结构调整、大规模重构 | 包名变更、模块拆分 |
| 引入新依赖、新服务 | 新增 Redis 缓存、引入新 SDK |
| 安全/权限相关的代码修改 | 修改权限校验逻辑、新增角色 |
| 需求本身不清晰 | 用户描述模糊，需要确认 |
| 无法用一句话说清 diff 时 | 变更逻辑复杂，影响范围广 |

## 豁免条件

以下情况可跳过 Plan 直接执行：

| 豁免条件 | 示例 |
|----------|------|
| 用户明确说"直接改"、"不用规划" | 用户主动豁免 |
| 单文件小改动 | 修复 bug、调整配置、修改文案 |
| 变更范围明确且影响小 | 修改一个常量值 |

## 工作流程

```
接收需求 → 需求拆分细化 → 分析影响范围 → 输出 Plan → 用户确认 → 触发 /sdc-spec 或直接执行
```

### 详细步骤

1. **接收需求**
   - 理解用户意图
   - 判断需求是否清晰完整

2. **需求拆分细化**（关键步骤）
   - 根据项目架构和已有功能，引导用户逐步拆分细化需求
   - 识别模糊点，向用户提问确认
   - 将一句话需求拆解为可执行的子需求

3. **分析影响范围**
   - 搜索相关代码文件
   - 识别受影响的模块
   - 评估风险等级

4. **输出 Plan**
   - 按输出格式输出结构化 Plan
   - 等待用户确认

5. **用户确认后**
   - 确认 → 自动调用 `/sdc-spec` 创建 Spec 文件
   - 豁免 → 跳过 Spec 直接实现
   - 修改 → 调整 Plan 后重新确认
   - 拒绝 → 终止流程

## 需求拆分细化流程

当用户输入一句话需求时，主 CC 应主动引导拆分，而非直接输出 Plan。

### 拆分策略

```
一句话需求 → 识别模糊点 → 提问确认 → 拆解子需求 → 确认完整性 → 输出 Plan
```

### 拆分维度

| 维度 | 引导问题 | 示例 |
|------|----------|------|
| **业务场景** | 这个功能在什么场景下使用？ | "用户管理" → 是后台管理还是用户自助？ |
| **用户角色** | 哪些角色会使用？ | "查看订单" → 管理员看全部，用户只看自己的？ |
| **数据范围** | 涉及哪些数据？ | "导出报表" → 导出哪些字段？什么格式？ |
| **操作类型** | 需要哪些操作？ | "商品管理" → 增删改查都需要？还是只读？ |
| **交互方式** | 用户如何操作？ | "筛选功能" → 下拉筛选还是搜索框？ |
| **边界条件** | 有什么特殊情况？ | "支付功能" → 支付失败怎么处理？退款呢？ |
| **权限控制** | 谁能做什么？ | "审批功能" → 谁能发起？谁能审批？ |
| **已有功能** | 项目中是否已有类似功能？ | "新增通知" → 已有消息中心，是否复用？ |

### 拆分示例

**用户输入**：「做一个订单管理功能」

**主 CC 引导拆分**：

```markdown
## 需求拆分

### 已明确
- 功能：订单管理

### 需确认
1. **业务场景**：这是后台管理员的订单管理，还是用户端的订单查看？
2. **操作范围**：需要哪些操作？
   - [ ] 查看订单列表
   - [ ] 查看订单详情
   - [ ] 修改订单状态
   - [ ] 取消订单
   - [ ] 删除订单
   - [ ] 导出订单
3. **数据范围**：订单包含哪些信息？（商品、金额、状态、收货地址...）
4. **权限控制**：哪些角色可以操作？管理员和普通用户看到的订单范围是否不同？
5. **已有功能**：项目中是否已有商品管理、用户管理等相关模块？

请确认或补充以上内容，我将输出详细的 Plan。
```

### 拆分原则

- **不假设**：不确定的内容必须向用户确认，不自行假设
- **参考现有**：搜索项目中已有功能，建议复用而非重复开发
- **逐步细化**：先确认大方向，再细化具体细节
- **控制提问数**：每次最多 3-5 个问题，避免用户疲劳
- **提供选项**：给出可选答案，降低用户回答成本

## 输出格式

执行任何操作前，先输出以下结构的 Plan：

```markdown
## 需求复述
[一句话描述要做什么]

## 影响范围
- 涉及文件：[列出所有将被修改的文件]
- 影响模块：[列出受影响的模块]
- 数据库变更：[是否涉及表结构变更]
- 依赖变更：[是否引入新依赖]

## 修改步骤
1. [步骤1 — 具体到文件和操作]
2. [步骤2]
3. [步骤3]

## 风险评估
- 高风险：[如有，说明原因和应对措施]
- 中风险：[如有，说明原因和应对措施]
- 低风险：[如有，说明原因]

## 验收标准
- [具体可验证的交付条件1]
- [具体可验证的交付条件2]

## 预估工作量
- 后端：[文件数/接口数]
- 前端：[页面数/组件数]
- 数据库：[表数/字段数]
- 测试：[用例数]
```

## 后续流程

**Plan 确认后自动调用 Spec**：

用户确认 Plan 后，AI 应自动调用 `/sdc-spec` 命令创建 Spec 文件，无需用户再次手动触发。

```
用户说"确认"、"好的"、"可以"、"继续"等确认词
    ↓
AI 自动调用 /sdc-spec
    ↓
创建 .claude/specs/<feature>/ 下的四个文件
```

> **注意**：如果用户明确说"直接改"、"不用规划"等豁免词，或任务明显属于单文件小改动（修复 bug、调整配置），则跳过 Spec 直接实现。

## Plan 评审要点

输出 Plan 后，主 CC 应自检以下要点：

- [ ] 需求复述是否准确理解了用户意图
- [ ] 影响范围是否完整（有无遗漏文件/模块）
- [ ] 修改步骤是否有序、可执行
- [ ] 风险评估是否合理（高/中/低分级正确）
- [ ] 验收标准是否具体可验证
- [ ] 是否需要数据库变更（需通知 sda-db-implementer）
- [ ] 是否需要新依赖（需评估兼容性）

## 注意事项

- Plan 是承诺，不是猜测：不确定的内容标注为「待确认」
- 修改步骤要具体到文件和操作，避免模糊描述
- 风险评估要给出应对措施，而非只列出风险
- 验收标准必须可验证，避免「功能正常」等模糊描述
- Plan 确认后不可随意扩大范围，超出范围需重新 Plan
